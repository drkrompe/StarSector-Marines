# 2026-05-22 — Comms officer narrator + briefing-room UI rebuild

Follow-up to [`2026-05-21-3.md`](2026-05-21-3.md) — that session left
patron archetypes byte-defined-but-unused and offers without an expiry
mechanic. This session takes the mission-select surface from "3-column
intel screen with a stubbed flavor text" to a **briefing-room comms
console** with composable officer voice layering, real per-day offer
expiry, and a derived officer mood that reacts to how the company is
actually doing.

The reframing is load-bearing: the player no longer reads patron prose
directly — every brief comes through the contract comms officer (Castle
from MWMercs as the spiritual reference). See
[[project-comms-officer-narrator]] memory for the design intent and
[`../campaign/contracts.md` §"The comms officer is the narrator"](../campaign/contracts.md)
for the written design.

## What landed

### Content pipeline migration — patron + industry templates as JSON

```
mod/data/marines/patron_briefings.json
mod/data/marines/industry_missions.json
src/main/java/com/dillon/starsectormarines/ops/IndustryMissionTemplates.java
src/main/java/com/dillon/starsectormarines/campaign/PatronBriefingTemplates.java
```

Templates now live in `mod/data/marines/*.json`, loaded via the standard
`Global.getSettings().loadJSON(path, withMods=true)` route. Lazy-load +
`parse(JSONObject)` + `loadForTest()` seam, matching the lineage already
set by `PatronBriefingTemplates`. Followed by archetype byte populated
at house-seed time and surfaced through the briefing composer.

### Comms-officer voice layering — composables, not cartesian

```
mod/data/marines/comms_officer_voice.json
src/main/java/com/dillon/starsectormarines/campaign/
  OfficerMood.java              DESPERATE / GREEN / STEADY / SEASONED
  OfficerMoodReader.java        currentMood() — Sector-gathering + bucket()
  CommsOfficerVoice.java        prefix/suffix/summary loader
  BriefingComposer.java         compose(archetype, mood, contractId, ...)
  CommsOfficerSummary.java      renderOverview / renderForClient — per-day stable
```

Three independent axes:

- **Patron archetype** (body) — in `patron_briefings.json`, archetype-pure
- **Officer mood** (prefix + optional suffix) — in `comms_officer_voice.json`, mood-pure
- **Officer summary** (mission-select header line) — `summary.{overview, client}`
  pools inside the same voice file; nested under mood

Composed via `BriefingComposer.compose(archetype, mood, contractId, ...)`
into `prefix + body + (optional suffix)`. All picks deterministic from
the contract id so save/load and re-renders produce the same text.

Patron prose rewritten as third-person officer dispatches — "officer
notes the rep was rushed" rather than the patron speaking directly.
All 18 templates touched.

### Offer expiry — data layer

```
src/main/java/com/dillon/starsectormarines/campaign/
  PatronArchetype.java                + (minOfferDays, maxOfferDays) per archetype
                                      + rollOfferWindowDays(Random)
  CampaignState.java                  + contractOfferExpiresTick[] column
                                      + contractDaysLeft(row, currentDay)
                                      + readResolve backfill for legacy saves (-1)
  ContractState.java                  + EXPIRED (appended last for save compat)
  systems/ContractLifecycleSystem.java + OFFERED-aging branch flips to EXPIRED
                                       when day >= offerExpiresTick
```

Per-archetype windows (e.g. TIME_RUSHED 2-3 days, ESTABLISHED 10-14)
land at `ContractGenerator` spawn time and feed the dossier days-left
bar in the new UI.

### Officer-mood derivation — real signals, not the STEADY stub

```
src/main/java/com/dillon/starsectormarines/campaign/OfficerMoodReader.java
```

`currentMood()` now reads:
- **Cash + ship count** via `Global.getSector().getPlayerFleet()`
- **Monthly net + debt trend** via `SharedData.getData().getPreviousReport()`
  (vanilla `MonthlyReport` — `totalIncome / totalUpkeep / debt / previousDebt`)
- **Captain count** via `MarineRosterScript.roster().activeCount()`
- **MRB rep** via `CampaignState.playerMrbRep` (dormant — see TODO at call site)

Precedence DESPERATE → SEASONED → GREEN → STEADY. Pure `bucket(...)`
function on primitive inputs keeps 18 unit tests free of Sector mocks.

Runway bands anchored to `economy.md`'s "hard-fail early, steady-grind
mid" arc:

| Band | Trigger |
| --- | --- |
| DESPERATE | credits < 6 months upkeep OR two-month debt trend |
| SEASONED | 6+ captains AND positive net AND non-negative MRB AND 12+ months runway |
| GREEN | <3 captains OR <2 ships |
| STEADY | fallthrough |

Per-day cache in `OfficerHeaderWidget` keyed on sector-day prevents
hysteresis — without it, a single cargo sale could flip DESPERATE →
STEADY mid-frame as credits ticks across the threshold.

See [[feedback-paycheck-runway-window]] for the tuning intent and
[[project-reputation-as-commodity]] for the next planned brake.

## Bug fix worth remembering

`PatronBriefingTemplates` originally hardcoded `"templates"` as the
top-level JSON key, which broke when `MoodCheck` reused the same JSON
loader for `moods` schema. Split: dedicated `MoodCheck` validator.
Lesson — when a "JSON content loader" is the *third* consumer that
wants subtly different schema, the per-schema validator is the cleaner
shape; one parameterized loader trying to handle both is the wrong
abstraction.

Also: a stale-render bug in the expanded dossier card. The card was
reading from the cached missions list (the Mission instance captured
at layout time), not `ctx.getSelectedMission()`. Salvage adjustments
produce a new Mission instance with the same id, so the cached
reference fell behind. Fix: `Mission render = isExpanded ? expanded : m;`
in `layoutDossierStack` so the expansion reads fresh state every
re-layout. Mission ref-equality observer in `MissionSelectScreen` is
what triggers the re-layout on every salvage tick.
