# 2026-05-17 — Captain meta-layer + dynamic mission generation

## What shipped (uncommitted)

Wired the long-dormant captain bookkeeping into a real progression loop. A
parallel session was on tilesets/map gen, so this change is scoped to
`marine/` + `ops/MissionResolver.java` + `ops/MissionOutcome.java` +
`ops/ResultsScreen.java` — zero touches in `battle/`.

### Files

```
M src/.../marine/MarineRosterScript.java   # advance() ticks injury recovery
M src/.../marine/Rank.java                  # +xpToNext() with doubling curve
M src/.../ops/MissionOutcome.java           # +promotedTo (nullable Rank)
M src/.../ops/MissionResolver.java          # wipe→roll, traits, promotion
M src/.../ops/ResultsScreen.java            # promotion banner row
M mod/data/strings/strings.json             # resultsPromotionLabel/Fmt
```

### Functional changes

- **Injury recovery**: INJURED captains auto-flip to ACTIVE when the sector
  clock passes `injuredUntilDay`. Was a no-op TODO in `MarineRosterScript.advance`.
  Adds a "Returned to active duty" commendation line.
- **XP→promotion**: `Rank.xpToNext()` is the per-tier threshold (250 → 500 →
  1000 → 2000 → 4000 → 8000 → 16000; GENERAL = MAX_VALUE). `MissionResolver.apply`
  loops while crossing thresholds — cascading promotions in one mission work.
  `MissionOutcome.promotedTo` is non-null when crossed; `ResultsScreen` shows
  it. Adds a "Promoted to {rank}" commendation per step.
- **Commendation log**: appends on victory ("Led successful op — {name}"),
  KIA ("Killed in action — {name}"), wipe-survived ("Survived FoB overrun
  — extracted with serious wounds"), and per promotion. Format: `Day N: ...`.
- **Wipe→roll**: total wipe used to be deterministic KIA. Now 60% KIA / 40%
  INJURED with 45-day recovery, seeded deterministically off
  `(captain.id, mission.id)` so the result doesn't drift between
  compute calls. Captured branch (10%) deferred until a `CAPTURED` Status
  enum lands.
- **`Trait.NATURAL_LEADER`**: ×1.5 XP multiplier on victory. Reads in
  `MissionResolver.compute`.
- **`Trait.FIELD_MEDIC`**: −25% marines-lost (rounded down). Reduces cargo
  decrement AND tightens the casualty XP malus threshold simultaneously.

### What didn't ship this session

- **`Trait.SCOUT`** — current briefing already shows enemy air support
  unconditionally; SCOUT would need to either hide-then-reveal that info, or
  expose new pre-rolled enemy-ground composition data on `Mission`. Deferred.
- **`Trait.LOGISTICS_CHIEF`** — `Rank.squadCap()` exists but isn't consumed
  anywhere in the deploy flow yet. Wiring a real marines-cap into briefing/
  battle is a bigger lift than this slice. Deferred.
- **Combat-touching traits** (`VETERAN`, `SIEGE_SPECIALIST`, `COMBAT_ENGINEER`)
  — intentionally held until the tileset/map session merges and the combat
  sim is stable to extend.

## Notes / accepted tradeoffs

- **`MissionResolver.compute` is no longer strictly pure** in the
  documented "same inputs same outcome" sense — wipe-fate is rolled there.
  Determinism is preserved via the captain+mission seed, so a re-compute
  yields the same fate. Updated the class javadoc.
- **Wipe-survived commendation uses a duration heuristic** (long INJURED
  → 45-day timer ≈ wipe-survived). Slightly fragile if injury durations
  change; cheaper than adding a `wipeSurvived` boolean to MissionOutcome.
- **No tests added** — existing test directory is for tileset code from the
  parallel session. Resolver math is exercised end-to-end via play; speculative
  unit tests skipped.

## Candidates for next session

In rough priority order:

1. **Squad cap wiring** — make `Rank.squadCap()` actually limit the marines
   deployed at briefing time. Unlocks `LOGISTICS_CHIEF`. Probably surfaces
   as a "max marines: N" row in the briefing transport section.
2. **SCOUT reveal** — pre-roll enemy ground composition in `MissionGenerator`,

   SCOUT-trait captain is selected.
3. **Bridge view / captain detail panel** — currently no way to read the
   commendations list a captain accumulates. The bridge view is the natural

4. **CAPTURED status + ransom mission** — finishes the 60/30/10 wipe-roll

5. **Combat-touching traits** — `VETERAN` (squad-wide HP/accuracy buff),
   `SIEGE_SPECIALIST` (bonus vs turrets), `COMBAT_ENGINEER` (vehicle perks)
   — once the tileset session merges.

## Personal note

The roster→combat feedback loop was the missing keystone — captains finally
*matter*. Even with just NATURAL_LEADER + FIELD_MEDIC wired, picking the
right captain has a real downstream effect. The wipe→roll is the design
swing I'm most curious about in playtest — does "60% chance you really lost

## Second pass — Dynamic mission generation from planet intel (uncommitted)

After the captain slice committed, expanded on the planet-side: the
generator used to be pure seeded RNG with no awareness of *what* was on the
target. Now it reads the market and emits missions per industry. Same
ship-then-optimize cadence: ground each piece in real planet state before
worrying about polish.

### New packages

```
src/.../ops/intel/                    — market-reading data layer
  DefenseLevel.java                   — 5-tier defense rating (enum)
  IndustryEntry.java                  — small snapshot DTO per industry
  PlanetIntel.java                    — frozen snapshot of a planet's market
  IntelReader.java                    — pulls PlanetIntel from a PlanetAPI
  MissionArchetype.java               — pre-baked (type, name, flavor) template
  IndustryMissionCatalog.java         — industry id → list of archetypes

src/.../ops/mission/story/            — hand-authored story missions
  StoryEligibilityContext.java        — bundle of planet+client+intel+roster+seed
  StoryMissionDef.java                — interface (id, isEligible, build)
  StoryMissionRegistry.java           — static list of defs + eligibleFor
  VeteransJobStory.java               — first concrete demo def

src/.../ops/MissionSource.java        — GENERATED vs STORY enum
```

### Modified

```
M src/.../marine/MarineRoster.java          # +completedStoryIds Set
M src/.../ops/Mission.java                  # +source, +targetPlanetName, +targetIndustryId
M src/.../ops/MissionGenerator.java         # rewrite: industries→missions + story prepend
M src/.../ops/MissionOutcome.java           # +missionId, +source, +type, +target*
M src/.../ops/MissionResolver.java          # industry disruption writeback + story-completion
```

### Functional changes

- **DefenseLevel** is a weighted sum of military structures (Patrol HQ ladder),
  orbital fortifications (Orbital Station ladder), Ground Defenses /
  Heavy Batteries / Planetary Shield, with a stability nudge. Bucketed into
  UNDEFENDED / LIGHT / MODERATE / HEAVY / FORTRESS.
- **MissionGenerator** now walks `intel.industries` and emits one archetype
  per non-disrupted industry, capped at 6 missions. Risk derives from
  DefenseLevel + mission type (Sabotage/Extraction drop a tier for stealth).
  Payout = size × risk × type × noise, rounded to 500.
- **IndustryMissionCatalog** maps 15 vanilla industry ids to 1–2 archetypes
  each — refinery sabotage reads as "Cripple the Refinery," patrol HQ
  assault as "Overrun the Patrol Post," etc. Flavor is industry-specific
  prose, not the old generic pool.
- **Industry damage counter + disruption**: on victory of an industry-targeted
  mission, `MissionResolver.applyIndustryDisruption` looks up the planet by
  name in `Global.getSector().getEconomy().getMarketsCopy()` and charges a
  per-industry damage counter held in the market's `MemoryAPI` under
  `$marines_industry_damage_<industryId>`. Damage per type: Sabotage +2,
  Raid +1, Assault +2, Extraction 0. Threshold 4 fires `setDisrupted` for
  the type-specific duration (Sabotage 60d, Raid 30d, Assault 90d) and
  resets the counter. So one sabotage softens a factory, two flatten it —
  matching vanilla's "sustained pressure" gate without forcing the player
  to orbit and wait. Disrupted industries get filtered out of the next
  visit's mission list — natural feedback loop.
- **Story missions**: source-tagged Missions prepended by
  `StoryMissionRegistry.eligibleFor`. Completion tracked via
  `MarineRoster.completedStoryIds` (xstream-persisted alongside captains).
  One demo def — "The Veteran's Job" at Independent clients once any
  captain reaches Corporal+.

### Notes / tradeoffs

- **No moves, only additions.** `Mission.java`, `MissionGenerator.java`, and
  `MissionResolver.java` stayed in `ops/` rather than moving into
  `ops/mission/`. Moving them would cascade import changes across BriefingScreen,
  MissionSelectScreen, BattleScreen, TacticalMapPanel, ResultsScreen,
  MarineOpsContext, etc. — too much churn for too little gain right now.
- **MissionOutcome got chunky** — 16 constructor args. Worth factoring into a
  builder if we add one more field. Held off until then.
- **Disruption on Assault is harsh** (90 days) — playtest may show this
  needs softening. Easy single-constant change.
- **Story registry uses static array** — no service-loader / runtime registration.
  Fine for the first few defs; convert when this file becomes a merge hotspot.
- **No on-victory hooks on StoryMissionDef yet** — the demo doesn't need
  one. Add when the second def needs to inject roster side effects (free
  captain, faction rep swing, follow-up unlock).

### Candidates for next session

1. **Disrupted-industry visibility on the briefing/intel screen** — "Refinery
   (disrupted)" badge so the player can read the consequence of their last op.
2. **Defense-level surfacing on the intel screen** — current `PlanetIntelPanel`
   shows type/faction/population/conditions; adding `defenseLevel.displayName()`
   in `defenseLevel.color()` is a one-line win.
3. **Battle layer reads DefenseLevel** — `BattleSetup` could pull the value
   off Mission's target planet and scale enemy unit count. This is where
   the meta + tactical layers actually merge.
4. **Squad cap wiring** — still deferred from the first pass. `Rank.squadCap()`
   exists but isn't consumed. Unlocks `LOGISTICS_CHIEF`.
5. **Second story def with an on-victory hook** — forces the StoryMissionDef
   interface to grow the `onVictory(outcome)` method, validates the pattern.

## Personal note (second pass)

Hit four substantial chunks in one session run — intel package, generator
rewrite, disruption loop, story branch — and the build stayed green through
each step. The single-call `industry.setDisrupted(...)` is the cheapest
*and* most-leveraged line of code in the whole session: one line, and now
every successful sabotage permanently (well, for 60 days) changes what a
planet offers. That's the texture I was looking for.

## Third pass — Heavy mech unit (uncommitted)

Wired `mod/graphics/battle/heavy-mech.png` into the battle layer as a new
defender elite. Lore peg: the in-game Heavy Armaments commodity description
calls out "mechs, tanks, hovercraft" — so the mech only shows up on planets
whose markets produce (Heavy Industry, Orbital Works) or demand (Ground
Defenses, Heavy Batteries) that commodity.

### Files

```
M src/.../battle/UnitType.java              # +HEAVY_MECH, +FrameLayout enum, +renderScale
M src/.../battle/BattleSetup.java           # +enemyHasHeavyArmor param + back-compat overloads
M src/.../ops/BattleScreen.java             # 8-way facing path, renderScale on draw
M src/.../ops/BriefingScreen.java           # planetHasHeavyArmaments() helper passed through
```

### Functional changes

- **New UnitType.HEAVY_MECH**: 90 HP / 4 dmg / 1.4 speed / 0.40 acc / 0.6 cooldown /
  26 range. Roughly 4× marine DPS and ~3.5× marine HP — one mech is worth
  ~3 marines in raw stats. `renderScale = 1.6f` so it visually overhangs
  adjacent cells and reads as a walker, not infantry.
- **`FrameLayout` enum on `UnitType`**: encodes the per-sheet frame-indexing
  convention. `WNES_WEAPON_UP` (default) is the 7-frame marine convention.
  `EIGHT_WAY_NO_WEAPON_UP` is new: 0=W, 1=NW, 2=SE, 3=S, 4=SW, 5=NE, 6=E.
  The mech sheet has no N frame; we fall back to NW for north-facing rather
  than synthesizing a flipped frame (mech is left/right asymmetric — chaingun
  arm vs. rocket-pod arm).
- **Renderer 8-way path**: `BattleScreen.renderUnitSprite` branches on
  `frameLayout`. New `EightWayFacing` enum + `computeEightWayFacing` +
  `pickFrameEightWay`. Octant bucketing uses `tan(22.5°) ≈ 0.414` so each
  octant is a clean 45° wedge.
- **`renderScale` applied at draw**: `targetH = unitSize * u.type.renderScale`
  with width preserved by aspect. Existing units stay at 1.0×.
- **`BattleSetup` defender mix**: when `enemyHasHeavyArmor`, slot 0 (highest-
  cover cell per `pickDefensiveCluster`) becomes a `HEAVY_MECH` instead of a
  red-marine elite. Caps at one mech per battle — readable threat, easy to
  tune up later. Two red-marines + 9 militia still fill the rest.
- **BriefingScreen probe**: `planetHasHeavyArmaments(targetPlanetName)` walks
  the economy via `Global.getSector().getEconomy().getMarketsCopy()` (same
  pattern `MissionResolver.applyIndustryDisruption` uses) and returns true
  if any of HEAVYINDUSTRY / ORBITALWORKS / GROUNDDEFENSES / HEAVYBATTERIES
  exists on the target market. Threaded through both
  `createPlaceholder(seed, manifest, enemyHasHeavyArmor)` and
  `createSabotage(seed, manifest, enemyHasHeavyArmor)`; the old 2-arg
  overloads default the flag to false.

### Notes / tradeoffs

- **No corpse sprite**: dead mechs vanish. A smoking-wreck doodad on death
  would read better — deferred. The existing `SmokingWreck` infra (currently
  turret-only) is a natural home for that.
- **No chaingun/rocket weapon distinction**: the mech fires using its base
  `UnitType` stats (single-shot, line tracer) — same code path as militia /
  red-marine defenders. The 4 dmg / 0.6 cooldown numbers give it
  chaingun-feel cadence, but the chainguns + rocket pods on the sprite are
  visual flavor only for now. A real burst/rocket profile probably wants
  defender-side weapons split off from `MarineWeapon` (which is marine-only).
- **Renderer fallback for the missing N frame is NW, not synthesized**: NE
  would also be plausible. Picked NW arbitrarily; both look fine in playtest
  since the mech rarely faces dead-N (path/target deltas almost always have
  some E/W component).
- **One mech, fixed slot**: deliberately the cover-best cell so the mech
  ends up commanding a sightline. If playtest reads as "the mech is always
  in the same corner," easy follow-up is to randomize which of the top-3
  cells gets the swap.

### Candidates for next session

1. **Mech wreck doodad** — extend `SmokingWreck` (or a new `LargeWreck`) so
   dead mechs leave a hulk that smokes and blocks the cell. Higher-value
   feedback than "mech disappears."
2. **Defender heavy weapons**: split a `DefenderWeapon` (or generalize
   `MarineWeapon` to `GroundWeapon`) so chainguns and rocket pods get their
   own tracer / sound / projectile profile. Closes the visual loop on the
   mech's actual armament.
3. **Mission briefing hint** — surface "Heavy armor expected" or similar on
   the briefing screen when `enemyHasHeavyArmor` is true, so the player can
   read the threat before accepting.

## Fourth pass — Mech chassis weapons (uncommitted)

Followed the third pass with the actual chassis-weapon system. The mech now
fires three concurrent weapon tracks (chainguns + SRM pod + LRM artillery)
instead of single-shot line tracers. Each weapon runs in its own range band
on its own cooldown so the mech *reads* as mayhem — brrt at close range,
SRM wave when targets cluster mid-close, LRMs lobbed at far targets.

### New files

```
src/.../battle/MechWeapon.java          — enum: CHAINGUN / SRM_POD / LRM_ARTILLERY
src/.../battle/MechLoadoutState.java    — per-unit mutable state bag (ammo, cooldowns, burst trackers)
```

### Modified

```
M src/.../battle/UnitType.java                  # HEAVY_MECH attackRange 26→40 (LRM reach)
M src/.../battle/Unit.java                      # +mech (MechLoadoutState, null on non-mechs)
M src/.../battle/ShotEvent.java                 # +mechWeapon field, +4-arg ctor
M src/.../battle/BattleSimulation.java          # +fireMechWeapon, +advanceMechWeapons pass
M src/.../battle/BattleSetup.java               # HEAVY_MECH spawn assigns defaultLoadout
M src/.../battle/ai/CombatantBehavior.java      # mech branch — tryFireMechWeapons; mech doesn't reposition mid-stream
M src/.../ops/BattleScreen.java                 # mech projectile sprite cache + loader + render branch + fire/impact SFX
```

### Functional changes

- **`MechWeapon` is a separate enum from `MarineWeapon`**. Same field shape
  (range/damage/accuracy/cooldown/burst/projectile/sound/impact) but
  conceptually different — chassis hardpoints, not handheld gear. Separation
  is deliberate so future MechWeapon variants (AC/20 swap-in, PPC variant)
  don't pollute the marine weapon list and the swappable-chassis-loadout
  pivot lands cleanly.
- **`MechLoadoutState` carries per-unit firing state** — chaingun cooldown +
  burst trackers, SRM cooldown + ammo + salvo trackers, LRM cooldown + ammo.
  Set on `Unit.mech` at spawn. Null for non-mechs. The marine burst infra
  (`Unit.burstRemaining` / `burstTimer` / `burstTarget`) is left alone —
  mechs don't carry MarineWeapons so the fields stay clean for marines.
- **Three concurrent firing tracks** in `CombatantBehavior.tryFireMechWeapons`:
  - Chaingun: target within 22 cells + off cooldown → fire one round, queue
    the rest of the 8-round burst.
  - SRM pod: target within 18 cells + off cooldown + ammo > 0 → fire one
    rocket, queue the rest of the 4-rocket salvo.
  - LRM artillery: target within 40 cells + off cooldown + ammo > 0 + OUTSIDE
    chaingun range → lob one Pilum. The chaingun-exclusion gate stops the
    mech from wasting LRMs on close targets it's already brrt-ing.
  All three can trigger on the same tick.
- **`BattleSimulation.advanceMechWeapons`** drains queued chaingun bursts +
  SRM salvos at the per-weapon `burstSpacing` and ticks down per-weapon
  cooldowns. Mirrors `advanceBursts` for marines but reads from
  `MechLoadoutState`. Salvo target is locked at salvo start so a 4-rocket
  spread doesn't smear across multiple enemies.
- **Renderer dispatch**: `ShotEvent.mechWeapon` is the source tag. The
  projectile-sprite cache + load step is parallel to the marine primary
  path. Fire SFX: chaingun = `chaingun_fire` (volume 0.65 to stop 8-round
  bursts from drowning everything else), SRM = `annihilator_fire`, LRM =
  `pilum_lrm_fire`. HE-impact mech rounds (SRM / LRM) also play the shared
  `marines_explosion` clip on arrival; chaingun rounds are kinetic, no
  extra audio.
- **Mech doesn't reposition mid-fire** — the CombatantBehavior mech branch
  walks the path forward but doesn't roll the `REPOSITION_CHANCE` sidestep
  the marines use. A walking mech reads as a slow brawler advancing, not
  an infantryman sidestepping between shots.
- **HEAVY_MECH base `attackRange` raised 26 → 40** to match LRM reach so
  target acquisition reaches across the grid. Base `attackDamage` /
  `attackCooldown` are unused on mechs (weapons read from MechLoadoutState)
  but kept non-zero as a defensive fallback if `mech == null` somehow.

### Stats (per shot, before cover reduction)

| Weapon | Range | Dmg | Acc | Cooldown | Burst | vsTurret | Ammo |
|---|---|---|---|---|---|---|---|
| Chaingun | 22 | 2.0 | 55% | 0.50s | 8 @ 60ms | 0.4× | infinite |
| SRM Pod  | 18 | 7.0 | 55% | 4.00s | 4 @ 100ms | 2.0× | 6 salvos |
| LRM Arty | 40 | 22  | 50% | 5.00s | 1 | 2.5× | 4 shots |

Full chaingun burst lands ~8.8 (with 55% per-round hit), full SRM salvo
~15.4 expected against infantry / 31 against a turret, LRM ~11 expected
against infantry / 27 against a turret. With chaingun firing roughly
continuously at close range, mech DPS at chaingun band is ~17 — a marine
deals ~0.7. So 1 mech ≈ 24 marines DPS in its sweet spot. The 90 HP and
the slow movement are what keep it from being unbeatable.

### Notes / tradeoffs

- **Salvo + burst share a "decision drain" model**: AI emits the first
  shot synchronously when triggering, then the sim drains the rest at
  per-weapon spacing. Same pattern marines use for SMG bursts.
- **`fireMechWeapon` duplicates ~30 lines from `fireShot`** (cover-reduce,
  apply damage, fall-back roll, emit ShotEvent). Refactoring to a shared
  helper that takes `(accuracy, damage, vsTurretMult, lifetime, source tag)`
  would dedupe — held off since the source-tag plumbing across ShotEvent's
  4-way mutually-exclusive nullable fields is its own bit of mess.
- **LRM gating to "outside chaingun range"** is a doctrine call, not a hard
  rule. Playtest may show the mech needs to fire LRMs at close targets too
  for the burst-damage moment; easy single-line change.
- **No swappable loadout picker yet** — every HEAVY_MECH gets
  `MechLoadoutState.defaultLoadout()`. The infrastructure is ready (Unit
  holds the state, weapons are independent enum entries) for a future
  briefing-screen mech loadout panel.

### Candidates for next session

1. **MechBehavior + role variants** — promote `tryFireMechWeapons` into
   its own behavior class, add `MECH_ASSAULT` / `MECH_SEARCH_DESTROY` /
   `MECH_FIRE_SUPPORT` role enum values, tune each variant's target
   priority + engagement-band weights. This is the doctrine layer the
   user was asking about.
2. **Mech captain stats UI** — display chassis loadout + ammo remaining
   on a future mech-detail panel (parallel to captain panel).
3. **Player-fielded mech path** — high cost (heavy armaments commodity?
   special acquisition?), one-per-mission slot. Briefing screen needs a
   "chassis" selector row.

## Fifth pass — Mech FX polish (uncommitted)

Pure polish round on top of the fourth-pass weapons. Playtest revealed
the chaingun was too quiet, the LRMs were too sterile, and the mech
vanished on death with no visual payoff. All four pieces shipped.

### Modified

```
M src/.../battle/MechWeapon.java                # +arcHeight, +hitSpread, +engineTrail fields; LRM tuned to salvo
M src/.../battle/MechLoadoutState.java          # +LRM salvo state + wreckSpawned flag; LRM ammo as salvos
M src/.../battle/UnitType.java                  # HEAVY_MECH deadSpritePath = heavy-mech-dead.png
M src/.../battle/BattleSimulation.java          # +spawnMechWrecks pass; fireMechWeapon applies hitSpread
M src/.../battle/ai/CombatantBehavior.java      # LRM queues a salvo instead of single-shot
M src/.../battle/fx/ImpactFx.java               # +spawnEngineTrail, +spawnMuzzleFlash public helpers
M src/.../ops/BattleScreen.java                 # parabolic-arc render, engine-trail emit, muzzle flash, vol 1.0, dead-mech renderScale
```

### Functional changes

- **Chaingun volume 0.65 → 1.0** on the per-shot SFX dispatch. The brrt
  cadence IS the point — it should dominate the audio mix.
- **LRM is now a 5-rocket salvo** at 110ms spacing (instead of single
  shot). Per-rocket damage stepped down (22 → 14) so full-salvo expected
  damage holds at ~35 vs. infantry. Cooldown lengthened (5s → 7s) and
  ammo recounted from "4 shots" to "3 salvos × 5 rockets = 15 rockets"
  to keep total per-mech firepower roughly steady.
- **`MechWeapon.hitSpread`** — endpoint-scatter on hit, in cells. Pure
  visual scatter around the locked target. LRM uses 1.5 cells so a salvo
  reads as artillery rain instead of 5 rockets stacking on one cell.
  Misses get baseline scatter PLUS hitSpread (so indirect-fire misses
  scatter wider than rifle misses).
- **`MechWeapon.arcHeight`** — parabolic visual arc, peaks at midpoint
  with height `arcHeight` cells. LRM = 5.0 cells so rockets visibly rain
  down over buildings. Bearing is computed from the analytical tangent
  of the parabola so the rocket noses up at launch and pitches down on
  descent. Chaingun + SRM keep linear trajectories (arcHeight = 0).
- **`MechWeapon.engineTrail`** — when true, the renderer emits a glow
  particle at the rocket's tail each render frame. Hot-orange tint
  (`ENGINE_TRAIL_COLOR = 0xFF9040`). Lifetime 0.25s; successive frames
  overlap into a fading exhaust streak. SRM + LRM both engine-trail;
  chaingun shells are kinetic, no trail.
- **Chaingun muzzle flash** — `ImpactFx.spawnMuzzleFlash(s.fromX, s.fromY)`
  in the per-shot loop. Hot additive 0.55-cell radius, 0.08s lifetime.
  Bright pop at the mech's center cell per round; the 8-round burst
  reads as a strobe. Skipped for rockets — the projectile sprite leaving
  the mount is its own launch animation.
- **Dead mech corpse** — `heavy-mech-dead.png` wired as
  `HEAVY_MECH.deadSpritePath`. The renderer's corpse pass now scales by
  `u.type.renderScale` so the hulk reads at 1.6× cell (matching the
  walker's visual footprint). `deathPoseIdx` % 4 picks the pose at death
  for variety.
- **Mech smoking-wreck** — new `BattleSimulation.spawnMechWrecks` tick
  pass, idempotent via `MechLoadoutState.wreckSpawned` so the wreck
  emits exactly once regardless of which code path landed the kill
  (chaingun crossfire, marine rocket, flyby strafe, debug damage). Wreck
  cell isn't flipped to walkable rubble (unlike turret death) — the
  hulk sprite occupies the cell and marines path around it.

### Stats post-tuning

| Weapon | Range | Dmg | Acc | Cooldown | Burst | Spread | Arc | Ammo |
|---|---|---|---|---|---|---|---|---|
| Chaingun | 22 | 2.0 | 55% | 0.50s | 8 @ 60ms | 0 | 0 | ∞ |
| SRM Pod  | 18 | 7.0 | 55% | 4.00s | 4 @ 100ms | 0 | 0 | 6 salvos |
| LRM Arty | 40 | 14  | 55% | 7.00s | 5 @ 110ms | 1.5 | 5.0 | 3 salvos |

### Notes / tradeoffs

- **Engine trail emit rate is render-frame-rate-dependent** — at 144fps
  the trail is denser than at 60fps. Tolerable because the particle
  lifetime is so short; if it ever looks wrong, gate emit by sim time.
- **Mech wreck doesn't open the cell to walking** — a marine standing on
  a mech-hulk cell would look wrong, but a marine BLOCKED by a hulk

  one-line swap.

  a separate in-progress map-gen refactor (untracked `Compound.java`,
  `MilitaryBaseFiller.java`, etc.) — not related to the mech changes.
  Mech branch compiles clean.

### Heat mechanic — recommendation for next pass

The user floated MechWarrior-style heat. I think this is the right
unifying mechanic *if* we want to scale up to multiple mech chassis
with very different weapon mixes. The shape would be:

- Each weapon adds heat on fire (chaingun small per round; SRM medium
  per salvo; LRM big per salvo).
- Heat dissipates at a steady cooldown rate (~2/sec).
- Heat capacity ~20; weapons disabled when heat ≥ capacity (overheat).
- AI fire prioritization becomes heat-aware: don't fire LRM if the
  resulting heat would lock out chainguns when a close target is incoming.
- Visual: mech sprite glows red / vents steam particle when heat ≥ 75%.

**Pro**: unifies the throttle. Cooldowns become per-shot heat costs;
overheat lockouts replace flat ammo caps for the "I shot too much" feel.
Pairs naturally with mech variants ("light mech: low heat cap, slow
weapons; heavy mech: high cap, fast hot weapons").

**Con**: another state machine + UI surface + AI rule. Bigger investment
than tuning numbers; only really pays off if there are multiple chassis
or a player-side mech to manage. As a fallback, the current per-weapon
cooldowns already gate DPS; heat is the upgrade, not a fix.

My instinct: hold heat until the second mech chassis or the player-mech
path lands. The rebalance above should make the existing single chassis
feel right in playtest; revisit heat when the system has more chassis
diversity to differentiate.

2. **Mech UI / debug overlay** — show the three weapon cooldowns + ammo

## Seventh pass — AoE rockets + physics-based impact (uncommitted)

Audit + fix pass. The user spotted that marine rockets damage at fire time
rather than at projectile arrival, and noticed neither rockets nor missiles
had splash damage or wall destruction. Both got rebuilt around a unified
`PendingDetonation` pipeline. "D-Day for your 100 marines" is the framing.

### New file

```
src/.../battle/PendingDetonation.java   — POJO: endpoint, lifetime, AoE radius, dmg, vsTurret, wallDmg, faction
```

### Modified

```
M src/.../battle/MarineSecondary.java          # +aoeRadius (1.5), +wallDamage (50)
M src/.../battle/MechWeapon.java               # +aoeRadius + wallDamage on every entry (0/0 for chaingun)
M src/.../battle/BattleSimulation.java         # +pendingDetonations + advancePendingDetonations + detonate
                                                # fireSecondary queues detonation instead of damage-at-fire
                                                # fireMechWeapon branches: AoE queue, kinetic stays damage-at-fire
```

### Functional changes

- **Marine rocket no longer damages at fire time.** `fireSecondary` rolls
  accuracy to determine the impact endpoint, queues a `PendingDetonation`
  with the weapon's `flightSec` lifetime, and emits the ShotEvent for the
  visual. The detonation pass applies damage on arrival — so a target who
  moves between launch and impact escapes the splash. This was the bug
  the user flagged.
- **Mech SRM + LRM moved to the same physics-based path.** Chaingun keeps
  the original kinetic damage-at-fire model (fast flight, single-target).
  The branch is `if (weapon.aoeRadius > 0)` inside `fireMechWeapon`.
- **AoE splash on detonation.** Every unit within `aoeRadius` of the
  endpoint with line-of-sight to the endpoint cell takes full weapon
  damage (cover DR applies; vsTurretMult applies to turrets). LOS check
  is the key feedback loop: marines hiding behind walls now actually
  benefit from the wall — the splash doesn't ignore terrain.
- **Friendly fire ON.** The AoE pass doesn't filter by faction. A marine
  who fires a rocket at a cluster will absolutely catch their own squad
  if they're inside the radius. The user explicitly wants this — adds
  weight to where you commit a rocket marine in the line.
- **Wall damage at endpoint cell.** `damageCell(endpointCx, endpointCy,
  weapon.wallDamage)` chews wall HP on arrival. `damageCell` already
  no-ops on walkable cells, so a rocket landing on ground just doesn't
  affect walls. Misses against units now still cost the *city*
  something — over the course of a battle the map degrades.

### AoE + wall stats

| Weapon | AoE radius | Wall HP per impact | Notes |
|---|---|---|---|
| Marine rocket (Annihilator) | 1.5 | 50 | 2 rockets flattens a 100-HP wall |
| Mech SRM (per rocket) | 1.3 | 25 | full 4-rocket salvo on one cell = wall down |
| Mech LRM (per rocket) | 2.0 | 60 | one rocket damages walls, two flattens; spreads across 5 cells thanks to hitSpread |
| Mech chaingun | 0 | 0 | kinetic, single-target, no structural |

### Notes / tradeoffs

- **Hit/miss roll still happens at fire time** — it only determines
  WHERE the rocket lands (target cell on hit, scattered on miss). The
  AoE math at impact decides who gets hurt. This means a precision hit
  → endpoint = target cell → target is at center of splash → max damage
  to them. A miss → endpoint scattered → target might be outside radius
  → escapes. Emergent and clean.
- **LOS check from endpoint cell to victim cell** uses Bresenham (cheap).
  A rocket exploding around a corner spares units behind the wall. This
  is the single most important rule for "rockets feel right" — without
  it, the splash makes cover meaningless.
- **No fall-back roll on AoE damage.** Fall-back fires on precision
  hits (rifle, mech chaingun) where "you're being targeted" is the
  read. AoE is chaotic — every unit in radius eats damage; nobody is
  uniquely "shot at." Skipping fall-back keeps the rocket as a chunky
  damage event without sending half the survivors retreating.
- **Pending detonation list is unbounded.** Worst-case = number of
  in-flight rockets at any moment. With 3 marines firing rockets every
  3s + one mech firing 5-rocket LRM salvos every 9s, the list peaks at
  maybe 10 entries. No cap needed.
- **The "marine fires rocket at his own feet" edge case is now
  consequential.** A marine in fall-back retreat firing rockets behind
  them could splash their own retreating squad. Not exercised today
  (AI doesn't fire while falling back), but the system supports it.
